2026-03-20

C++ 智能指针与内存管理

C++ 没有自动垃圾回收，手动管理内存极易产生泄漏或悬空指针。智能指针（Smart Pointer）通过 RAII（资源获取即初始化）机制，在对象生命周期结束时自动释放资源，是现代 C++ 推荐的内存管理方式。C++11 引入三种智能指针，均定义在 <memory> 头文件中：

  • std::unique_ptr：独占所有权，不可复制，只能移动。
  • std::shared_ptr：共享所有权，通过引用计数管理资源。
  • std::weak_ptr：弱引用，配合 shared_ptr 使用，解决循环引用。

std::unique_ptr —— 独占所有权

unique_ptr 对资源拥有唯一所有权，不能复制，只能移动。它轻量高效，几乎没有额外开销。

#include <memory>

// C++11 方式
std::unique_ptr<int> p1(new int(42));

// C++14 推荐使用 make_unique（更安全，避免显式 new）
auto p2 = std::make_unique<int>(100);

// 不能复制
// auto p3 = p1;                // 编译错误

// 可以移动（所有权转移）
auto p3 = std::move(p1);        // p1 现在为 nullptr

// 获取原始指针（谨慎使用）
int* raw = p2.get();

// 释放所有权并返回原始指针
int* released = p3.release();   // p3 变为 nullptr，需要手动 delete released

// 重置：释放当前管理的对象，可指向新对象
p2.reset(new int(50));

自定义删除器

unique_ptr 的删除器是类型的一部分，因此需要在模板参数中指定。

1
2
3
4
5
6

// 函数指针作为删除器
auto deleter = [](FILE* f) { if (f) fclose(f); };
std::unique_ptr<FILE, decltype(deleter)> filePtr(fopen("test.txt", "r"), deleter);

// 若删除器是无捕获 lambda，可直接使用（C++11 需函数指针，C++14 泛型 lambda 支持）
std::unique_ptr<int, void(*)(int*)> ptr(new int(10), [](int* p) { delete p; });

若删除器捕获了变量，则无法转换为函数指针，需使用 std::function 包装：

1
2
3

int x = 0;
auto lambda = [&](int* p) { delete p; std::cout << x; };
std::unique_ptr<int, std::function<void(int*)>> ptr(new int(10), lambda);

常用场景

  • 工厂函数返回独占对象：函数返回 unique_ptr 转移所有权给调用者。
  • 作为类成员：表示对象独占某资源。
  • 放入容器：容器存放 unique_ptr 时需使用移动语义（如 push_back(std::move(ptr))）。
1
2

std::vector<std::unique_ptr<int>> vec;
vec.push_back(std::make_unique<int>(1));   // 移动构造

std::shared_ptr —— 共享所有权与引用计数

shared_ptr 允许多个指针共享同一资源，内部维护一个引用计数，计数为 0 时自动释放资源。

初始化与引用计数 

1
2
3
4
5
6
7
8
9

auto sp1 = std::make_shared<int>(200);   // 推荐，只分配一次内存（控制块+对象）
std::cout << sp1.use_count();             // 1

auto sp2 = sp1;                            // 拷贝构造，引用计数 +1
std::cout << sp1.use_count();               // 2

sp2.reset();                                // 引用计数 -1
std::cout << sp1.use_count();               // 1
// sp1 离开作用域后计数归零，内存释放

重要：不要用同一个裸指针初始化多个 shared_ptr，否则会导致多个独立的控制块，造成重复释放。

1
2
3

int* p = new int(10);
std::shared_ptr<int> sp1(p);
std::shared_ptr<int> sp2(p);   // 危险！sp1 和 sp2 各自管理 p，会双重释放

自定义删除器

shared_ptr 的删除器类型擦除，无需在模板参数中指定。

1
2
3

std::shared_ptr<int> sp(new int[10], [](int* p) { delete[] p; });   // 数组删除器
// 或使用 std::default_delete
std::shared_ptr<int> sp2(new int[10], std::default_delete<int[]>());

获取原始指针 

1

int* raw = sp.get();    // 不增加引用计数，需确保 sp 未释放

线程安全说明

  • 引用计数的增减是原子操作，线程安全。
  • 但 shared_ptr 对象本身的读写（如多个线程同时修改同一个 shared_ptr 变量）不是线程安全的，需要外部同步。

std::weak_ptr —— 打破循环引用

weak_ptr 是 shared_ptr 的助手，不增加引用计数，用于观测资源是否存活，不拥有资源。它不能直接解引用，必须通过 lock() 获取一个有效的 shared_ptr 才能访问资源。

基本用法 

1
2
3
4
5
6
7
8

std::shared_ptr<int> sp = std::make_shared<int>(300);
std::weak_ptr<int> wp = sp;   // 引用计数仍为 1

if (auto locked = wp.lock()) {   // 获取 shared_ptr，若资源已释放则返回空
    std::cout << *locked;
}
std::cout << wp.expired();        // 返回 true 表示资源已释放
wp.reset();                       // 清空监测

解决循环引用

循环引用会导致 shared_ptr 计数永远不为 0，内存泄漏。典型场景是双向链表或父子互相指向。

1
2
3
4
5
6
7
8
9

struct Node {
    std::shared_ptr<Node> next;   // 问题所在！
    ~Node() { std::cout << "~Node\n"; }
};

auto a = std::make_shared<Node>();
auto b = std::make_shared<Node>();
a->next = b;
b->next = a;   // 循环引用，a 和 b 的引用计数均为 2，无法归零，析构函数不会调用！

解决方案：将其中一个成员改为 weak_ptr。

struct Node {
    std::weak_ptr<Node> next;   // 改用 weak_ptr
    int val;
};

auto a = std::make_shared<Node>();
auto b = std::make_shared<Node>();
a->next = b;   // 不增加引用计数
b->next = a;
// 离开作用域后，a 和 b 的引用计数正常归零，析构函数调用

返回管理 this 的 shared_ptr

若需要在类内部返回一个管理当前对象的 shared_ptr，直接使用 this 构造会导致多个控制块。应继承 std::enable_shared_from_this<T>，并调用 shared_from_this()。

1
2
3
4
5
6
7
8

struct Test : std::enable_shared_from_this<Test> {
    std::shared_ptr<Test> getShared() {
        return shared_from_this();   // 内部使用 weak_ptr 监测 this
    }
};

auto sp = std::make_shared<Test>();
auto sp2 = sp->getShared();   // sp2 与 sp 共享同一控制块，引用计数正常增加

注意：必须在已经存在 shared_ptr 管理当前对象的情况下才能调用 shared_from_this()，否则会抛出异常。

总结与建议

智能指针 特点 适用场景
unique_ptr 独占所有权，轻量，无额外开销 资源唯一所有，工厂函数返回值，容器元素
shared_ptr 共享所有权，引用计数，稍重 多个所有者，缓存，对象生命周期需动态管理
weak_ptr 不控制生命周期，观测 shared_ptr 资源 打破循环引用，缓存（弱回调），可选依赖

最佳实践：

  • 尽量使用 make_unique / make_shared，避免显式 new。
  • 优先选用 unique_ptr，只有在需要共享所有权时才用 shared_ptr。
  • 使用 weak_ptr 打破循环引用，但不要长期持有 lock() 返回的 shared_ptr。
  • 从智能指针获取裸指针仅用于与遗留 C API 交互，且必须确保智能指针未释放。
  • 不要将同一个裸指针交给多个独立的智能指针管理。
